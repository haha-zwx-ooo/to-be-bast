# ReentrantLock

上文讲了java对象内置锁:synchronized,这回说说显示锁ReentrantLock,同时讲讲二者的区别和使用场景

synchronized 是 JVM 层面通过监视器(Monitor)实现的，而 ReentrantLock 是通过 AQS(AbstractQueuedSynchronizer)程序级别的 API 实现。

ReentrantLock 在使用之前需要先创建 ReentrantLock 对象，然后使用 lock 方法进行加锁，使用完之后再调用 unlock 方法释放锁，具体使用如下：

public class LockDemo {
    
    Lock lock = new ReentrantLock();
    
    int index = 0;
    
    public static void main(String[] args) {
        LockDemo l = new LockDemo();
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 5; i++) {
                l.add();
            }
        });
        Thread t2 = new Thread(() -> {
            for (int i = 0; i < 5; i++) {
                l.add();
            }
        });
        t1.start();
        t2.start();
    }
    
    public void add() {
        // 加锁操作
        lock.lock();
        try {
            index++;
            System.out.println(index);
        } finally {
            lock.unlock();
        }
    }

}

# 区别

  • synchronized 可用来修饰普通方法、静态方法和代码块，而 ReentrantLock 只能用在代码块上。

  • synchronized 会自动加锁和释放锁，当进入 synchronized 修饰的代码块之后会自动加锁，当离开 synchronized 的代码段之后会自动释放锁

  • ReentrantLock 需要手动加锁和释放锁

  • synchronized 属于非公平锁，而 ReentrantLock 既可以是公平锁也可以是非公平锁。默认情况下 ReentrantLock 为非公平锁

  • ReentrantLock 可以使用 lockInterruptibly 获取锁并响应中断指令，而 synchronized 不能响应中断

  • synchronized 是 JVM 层面通过监视器(Monitor)实现的，而 ReentrantLock 是通过 AQS(AbstractQueuedSynchronizer)程序级别的 API 实现

  • lock条件锁Condition多维度await signal,synchronized只有wait 和 notify

# 性能

在低并发的情况下，synchronized的性能表现优于ReentrantLock，因为synchronized是JVM内置的关键字，JVM可以对其进行优化。

但在高并发的情况下，ReentrantLock的性能表现更好，因为它具有更细粒度的锁控制，能够更好地协调线程之间的并发访问。

总体来说，两者的性能差异并不明显，而且很难通过简单的基准测试来证明哪一个更快。在实际使用中，应该根据具体的场景来选择使用哪一个。
